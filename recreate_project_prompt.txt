02. WHAT WAS WRONG WITH V1 AND MUST NOT BE REPEATED
======================================================================

The old project had the following problems. Treat this list as a "never again" checklist.

1. Monolithic daily files like `day01_exercises.py` and `day01_solutions.py` containing many unrelated problems.
2. Duplicate week directories with conflicting naming schemes.
3. Ambiguous imports like `from day01_solutions import *`.
4. Tests relying on `sys.path` manipulation.
5. Root `pytest` failures caused by import collisions and duplicate test module names.
6. Committed generated artifacts such as `__pycache__`, `.pytest_cache`, `.coverage`, and `htmlcov`.
7. Status and summary documents that became stale and contradicted the real code.
8. Placeholder files committed as if the week was "done".
9. Fake or placeholder badges and URLs in README files.
10. Inconsistent package boundaries and multiple ways to import the same symbol.
11. Same symbol names reused in confusing ways across the same day.
12. Root configuration pointing at folders that were empty, renamed, or obsolete.
13. Too much repository noise compared with the amount of genuinely useful learning material.
14. Unnecessary dependencies or stray imports in beginner materials.
15. Incomplete weeks that still looked finished from the documentation.

If you are ever unsure between reproducing old behavior and choosing a cleaner structure, choose the cleaner structure.

======================================================================

Every week directory must be a real Python package.

  project/

Import policy:

1. Tests must use explicit imports.
2. Do not use wildcard imports.
3. Do not rely on the current working directory.
4. Do not rely on `sys.path` mutation.
5. Prefer full module imports in tests, for example:

   `from week01_fundamentals.solutions.day01.problem_01_calculate_sum import calculate_sum`

6. Day package `__init__.py` files may re-export symbols for convenience, for example:

   `from .problem_01_calculate_sum import calculate_sum`

but tests should still prefer direct problem-module imports for maximum clarity.

Public symbol collision rule:

Within a single `dayNN` package, exported public symbol names must be unique.

If two exercises would naturally define the same class name, rename them to stay package-safe.

Example:

- bad: two different files each exporting `Duck`
- good: `BehaviorDuck` and `WaterfowlDuck`
- good: `BaseDuck` and `ComposedDuck`

Self-contained problem rule:

If one exercise requires multiple classes or helper functions that belong to that one exercise, keep them in the same problem file.

Example:

- `problem_01_car_composition.py` may contain `Engine`, `Transmission`, and `Car`

That is correct because it is still one exercise file.

Shared helper rule:

Prefer self-contained problem files.
Create shared helper modules only if all of the following are true:

1. the helper is used by at least three files
2. the helper is conceptually stable
3. sharing it improves clarity
4. the helper does not hide the pedagogical point

If shared helpers are needed, place them in one of:

- `dayNN/_shared.py`
- `project/shared/`

Do not create random utility sprawl.

12. SOLUTION FILE TEMPLATE REQUIREMENTS
======================================================================

Each solution file must include:

1. complete implementation
2. correct type hints
3. only useful comments
4. no unnecessary cleverness
5. no giant explanatory essays inside the code

Preferred solution file skeleton:

```python
"""Reference solution for Problem 01: Calculate Sum."""

from __future__ import annotations


def calculate_sum(a: int, b: int) -> int:
    """Return the sum of two integers."""
    return a + b
```

Class-based example:

```python
"""Reference solution for Problem 03: Bank Account."""

from __future__ import annotations


class BankAccount:
    """Simple bank account model."""

    def __init__(self, owner: str, balance: float = 0.0) -> None:
        self.owner = owner
        self.balance = balance

    def deposit(self, amount: float) -> bool:
        if amount <= 0:
            return False
        self.balance += amount
        return True
```

15. PEDAGOGICAL GUARDRAILS
======================================================================

The rebuild must respect learning progression.

Do not leak advanced ideas too early.

Pedagogical rules:

1. Week 1 should not rely on OOP for exercise solutions.
2. Week 2 should not require class-heavy designs except where the topic naturally needs it.
3. Week 3 should introduce classes cleanly before any inheritance-heavy material.
4. Week 4 is where inheritance and polymorphism become central.
5. Week 5 is the first place for descriptors, metaclasses, advanced decorators, and reflection.
6. Week 6 is where design patterns become explicit.
7. Week 7 should focus on pragmatic design and testing, not just theory.
8. Week 8 should integrate everything in a coherent capstone.

Additional educational rules:

1. Do not use `dataclass` in learner-facing Week 1-3 exercises unless explicitly justified.
2. Do not use descriptors or metaclasses before Week 5.
3. Do not introduce design-pattern jargon before Week 6 unless it is unavoidable.
4. Keep early-week solutions simple and idiomatic.
5. Prefer standard library solutions over external libraries.
6. Avoid "smart" code that hides the learning point.

Difficulty progression inside a day:

Use problem numbering to signal progression:

- Problems 01-03: warm-up / foundational
- Problems 04-06: core practice
- Problems 07-08: harder application
- Problems 09-10: stretch or bonus

Not every day must have exactly 10 problems, but the progression should feel intentional.

Quality over quantity.

Recommended counts:

- Week 1-2: 8 to 10 problems per day
- Week 3-4: 6 to 10 problems per day
- Week 5-7: 4 to 8 deeper problems per day
- Week 8: milestone-driven instead of many isolated drills
