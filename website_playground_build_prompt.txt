WEBSITE PLAYGROUND BUILD PROMPT FOR A NEW CHAT

You are helping me build the web platform for this curriculum.

Read this entire prompt before creating files, choosing frameworks, proposing
architecture, or making UI decisions.

This is not a marketing site.
This is not a generic LMS.
This is not a static docs wrapper.
This is not a toy code runner.

This should become the main interactive surface for learning through the
`python-oop-journey-v2` curriculum:

- read theory
- understand concepts
- move through the curriculum in order
- solve exercises in the browser
- run code safely
- verify work clearly
- get hints without getting spoiled immediately
- reveal solutions intentionally
- track real progress
- return later without losing context
- eventually handle weekly projects and capstone work well

Assume I already own the domain for this product.

Assume the curriculum repository is the source of truth for content.
Do not turn the website into a manual content-entry system.

If some later weeks are not yet fully present in the current workspace, do not
invent fake curriculum content. Design the platform so new weeks can appear
naturally when the repo gains real material.

========================================================================
01. PRIMARY MISSION
========================================================================

Create the best possible interactive learning platform and coding playground for
the `python-oop-journey-v2` curriculum.

The website must feel like a serious product for learning Python and OOP:

- intentional
- trustworthy
- calm
- fast
- code-first
- learner-friendly
- maintainable

It must support these real jobs well:

1. A learner reading and understanding theory
2. A learner solving exercises in an interactive Python editor
3. A learner verifying work without getting lost
4. A learner resuming exactly where they left off
5. A maintainer syncing the site with the curriculum repository cleanly

========================================================================
02. PRODUCT STANDARD
========================================================================

This should feel closer to:

- an interactive textbook
- a coding practice product
- a personal learning cockpit
- a serious curriculum platform

than to:

- a blog
- a random docs site
- a SaaS dashboard template
- a decorative AI-flavored landing page with an editor bolted on

The final product should be something I would actually want to use as my main
way to work through the curriculum on my domain.

========================================================================
03. USER-FIRST EXPECTATION
========================================================================

When making tradeoffs, optimize for the learner's lived experience.

The learner should consistently feel:

- I know where to start
- I know what this page is for
- I know what to do next
- I know how to run my code
- I know how to verify my work
- I know where to get help
- I do not feel punished for making mistakes
- I can come back tomorrow without reorienting from scratch

If a design looks technically correct but increases confusion, slow navigation,
or emotional friction, it is the wrong design.

========================================================================
04. WHAT SUCCESS LOOKS LIKE
========================================================================

A successful first serious version should let me do this smoothly:

1. Open the site on my domain
2. Immediately understand the curriculum layout
3. See where I left off
4. Open a week
5. Open a day
6. Read theory in a pleasant format
7. Jump straight into the first exercise
8. Edit starter code in a strong browser editor
9. Run my code
10. Verify my solution
11. Read meaningful failure feedback
12. Use hints if stuck
13. Reveal the solution intentionally if needed
14. Mark progress and move on
15. Return later and resume cleanly

If any of these steps feels clumsy, hidden, brittle, or annoying, the product
is not good enough yet.

========================================================================
05. HARD REQUIREMENTS
========================================================================

These are non-negotiable.

1. The website must ingest curriculum content from the repo structure.
2. Do not manually duplicate the curriculum into random CMS records.
3. Theory docs, exercise stubs, solutions, tests, and project docs must remain
   traceable to the repo.
4. The playground must not execute arbitrary learner Python inside the web app
   process.
5. Code execution must run in an isolated sandbox with strict limits.
6. The UX must clearly separate:
   - read theory
   - run code
   - verify work
   - reveal help
   - reveal solution
7. The site must support a real desktop coding experience.
8. The site must support a strong mobile reading experience.
9. The architecture must be maintainable and testable.
10. The custom domain must be treated as a first-class deployment concern.

========================================================================
06. CORE PRODUCT PRINCIPLES
========================================================================

Use these principles everywhere.

1. Clarity over cleverness
2. Guidance over clutter
3. Fast navigation over decorative density
4. Trust over hype
5. Learning support over raw correctness theater
6. Calm focus over noisy gamification
7. Safe execution over fast unsafe shortcuts
8. Repo ingestion over manual content drift

========================================================================
07. PRIMARY PERSONAS
========================================================================

Design mainly for these personas.

Persona 1: The Daily Learner

- Works through the curriculum in order
- Wants a clean next-step experience
- Needs help without immediate spoilers
- Wants to feel steady progress

Persona 2: The Returning Learner

- Has been away for a few days
- Needs fast reorientation
- Wants to see:
  - where they stopped
  - what is incomplete
  - what to review next

Persona 3: The Maintainer

- Adds or updates curriculum content in the repo
- Needs ingestion to be reliable
- Needs failures to be precise, not mysterious

Persona 4: The Reviewer

- Wants to inspect the platform and see that it is thoughtfully engineered
- Values coherence, safety, and professionalism

========================================================================
08. MOST IMPORTANT UX OUTCOME
========================================================================

The website should reduce friction between:

- learning a concept
- trying code
- getting feedback
- moving forward

The site should not force the learner to translate raw repository structure into
their own mental model. That translation is the product's job.

========================================================================
09. LEARNING LOOP
========================================================================

The product should be built around this loop:

1. Preview today's scope
2. Read the theory
3. Open the first problem
4. Try code
5. Run code
6. Verify
7. Inspect feedback
8. Use hints if needed
9. Save progress
10. Continue to the next problem
11. End with a project or review checkpoint

This loop should feel obvious and smooth from the UI alone.

========================================================================
10. ONBOARDING REQUIREMENTS
========================================================================

The onboarding experience matters.

The first visit should answer:

- What is this site?
- What is the curriculum?
- Where should I start?
- Do I need an account?
- How do exercises work here?
- What is the difference between run and verify?

The first-time user experience should ideally include:

1. a clean landing view with curriculum overview
2. a clear "Start Here" action
3. a short explanation of:
   - theory pages
   - problem pages
   - run vs verify
   - hints vs solutions
4. a resume or continue-learning affordance when the user returns

Avoid a wall of generic onboarding slides.
Teach the product through use, not through filler.

11. HOMEPAGE REQUIREMENTS

It should function as a real learner dashboard.

It should include:

1. current progress summary

========================================================================
19. RUN VS VERIFY UX
========================================================================

The platform must make the difference between Run and Verify extremely clear.

Run:

- executes the learner code
- good for experimentation
- should show direct output and runtime errors

Verify:

- checks correctness
- should use a learner-friendly harness
- should explain pass/fail clearly

Show Solution:

- should be intentional
- should not be the easiest thing on the page to click

This distinction is fundamental to the product.

========================================================================
20. VERIFICATION FEEDBACK REQUIREMENTS
========================================================================

Verification feedback should feel helpful, not punishing.

Good output should include:

1. pass/fail summary
2. which check failed
3. what kind of issue occurred:
   - wrong return value
   - raised unexpected exception
   - missing edge case
   - output mismatch
4. enough detail to debug
5. a path to next help:
   - hint
   - related note
   - theory link

Do not dump raw pytest output at the learner unless intentionally placed in an
advanced debug view.

The default learner surface should be cleaner and more interpretable.

========================================================================
21. HINTS, HELP, AND SPOILER CONTROL
========================================================================

The site should help learners without collapsing straight into answers.

Preferred help ladder:

1. Hint 1: conceptual nudge
2. Hint 2: structural guidance
3. Hint 3: edge-case reminder
4. Solution reveal

The UI should make this ladder visible.

The site should support:

- common mistakes
- debugging tips
- why-this-works notes
- related theory references

The product should be good for learning, not just scoring.

========================================================================
22. SOLUTION REVEAL UX

1. do not auto-show solutions

3. allow re-hide
4. keep the learner code visible nearby

The UI should communicate:

- This is the reference solution
- Use it after a real attempt
- Compare behavior and structure, not only final output

========================================================================
32. CONTENT INGESTION STRATEGY
========================================================================

This is one of the most important parts.

Do not hardcode the curriculum into web components.

The ingestion layer should derive structure from repo conventions:

- `weekXX_topic/README.md`
- `dayNN_topic.md`
- `exercises/dayNN/problem_XX_*.py`
- `solutions/dayNN/problem_XX_*.py`
- `tests/dayNN/test_problem_XX_*.py`
- `project/README.md`
- `project/starter/`
- `project/reference_solution/`

It should:

1. scan the repo
2. build a normalized manifest
3. extract docstring metadata when possible
4. extract theory metadata from markdown structure
5. preserve source-path traceability
6. fail loudly in development on malformed content

Minimum manifest fields:

- week slug
- week title
- day slug
- day title
- theory doc path
- problem slug
- problem title
- difficulty
- exercise path
- solution path
- test path
- project metadata

It must be easy to add later weeks without redesigning the website.

4. deployment on a custom domain

========================================================================
38. SANDBOX EXECUTION REQUIREMENTS
========================================================================

Never run learner Python directly inside the web app process.

Execution must be:

1. isolated
2. resource-limited
3. time-limited
4. network-restricted
5. filesystem-restricted
6. auditable

Suggested flow:

1. web app sends code to backend
2. backend creates execution job
3. worker runs code in sandbox
4. output and verification results return
5. sandbox is destroyed

Resource constraints should include:

- CPU time limit
- wall-clock timeout
- memory cap
- process cap
- no unrestricted networking
- no host filesystem access

Do not compromise here.

========================================================================
39. MOBILE VS DESKTOP EXPERIENCE
========================================================================

The product should be honest about screen constraints.

Desktop:

- primary environment for serious coding
- strong editor and side-by-side workflow

Mobile:

- strong reading experience
- review, bookmarking, and light note-taking
- maybe light editing, but do not optimize mobile around full coding parity if
  it hurts desktop quality

The site should never feel broken on mobile, but it does not need to pretend a
phone is the ideal place for full project coding.
